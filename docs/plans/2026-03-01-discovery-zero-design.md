# Discovery Zero: 基于推理超图的自主数学发现系统

## 1. 愿景与动机

### 1.1 核心问题

人类数学家在数千年中积累的定理体系是**历史偶然**的产物——受限于直觉偏好、文化传承和认知局限。是否存在更高效的定理体系？是否存在人类从未发现的深层数学关系？

更根本地：能否构建一台**自主数学发现机器**——它像科学家一样思考（提出猜想）、像实验物理学家一样验证（做计算实验）、像纯数学家一样证明（写形式化证明）——并在这个循环中持续产出新的数学知识？

### 1.2 从 AlphaZero 获得的启示

AlphaZero 在围棋中证明了一个深刻的观点：从零开始的自我探索可以超越人类数千年积累的知识体系。我们将这一思路迁移到数学发现：

| AlphaZero (围棋) | Discovery Zero (数学) |
|---|---|
| 棋盘状态 | 推理超图（已知对象 + 命题 + 置信度） |
| 落子 | 构造新对象 / 进行推理 / 做实验 |
| 自我对弈 | 自主探索循环 |
| 胜负信号 | 发现并证明新定理 |
| ELO 提升 | 发现更深、更新颖、更高效的定理 |

### 1.3 通用数学发现，而非特定领域

系统架构完全**领域无关**。核心循环——**推理 → 实验 → 证明**——适用于数学的所有分支：几何、数论、代数、组合、分析等。不同领域的区别仅在于实验模块的具体形式（几何用坐标计算，数论用整数枚举，代数用具体群/环实例……），而推理超图、贝叶斯推断、探索策略等核心架构完全共享。

## 2. 核心思想

### 2.1 三种认知模式

数学发现本质上依赖三种不同的认知模式，对应三种不同的确信度：

| | Plausible Reasoning | Experimental Verification | Formal Proof |
|---|---|---|---|
| **态度** | "我觉得..." | "试试看..." | "必然如此" |
| **手段** | 类比、直觉、猜想、模式识别 | 数值计算、具体实例、归纳检验 | Lean 形式化证明、严格演绎 |
| **确信度** | 低 (0.3 ~ 0.7) | 中 (0.8 ~ 0.95) | 高 (~1.0) |
| **性质** | 创造性 | 经验性 | 确定性 |

这三种模式对应人类数学家的日常工作：在草稿纸上涂涂画画（plausible）、打开 Mathematica 算一算（experiment）、然后坐下来写严格证明（formal）。

### 2.2 LLM 作为数学家的大脑

系统以大语言模型为核心决策者。LLM 本身就是 plausible reasoning 的载体——类比、猜想、模式识别、证明策略规划都是大模型的原生能力。数值实验和 Lean 证明则是 LLM 可以按需调用的工具。

```
LLM (推理 + 决策)
 |
 |-- Lean         (形式化证明)
 |-- 数值计算      (实验验证)
 +-- 定理库        (记忆与检索)
```

LLM 自己判断在什么时候使用什么工具：
- 直觉很强 → 直接写 Lean 证明
- 不确定 → 先跑几个数值实验
- 完全陌生的领域 → 先大量实验探索，再形成猜想
- 需要已有结果 → 查询定理库

## 3. 推理超图：统一的知识表示

### 3.1 超图结构

所有推理活动——无论来自哪个认知模式——都统一表示为推理超图中的**超边 (hyperedge)**。

一条超边由一组**前提节点**出发，经过一系列**推导步骤**，得到一个**结论节点**：

```
超边 (Hyperedge) = {
    id:          唯一标识符,
    premises:    [P₁, P₂, ..., Pₖ],          -- 前提命题节点集合
    steps:       [step₁, step₂, ...],         -- 推导过程
    conclusion:  C,                            -- 结论命题节点
    module:      "plausible" | "experiment" | "lean",
    confidence:  float ∈ [0, 1]                -- P(C正确 | P₁...Pₖ都正确)
}
```

### 3.2 不同模块的超边内容

虽然格式统一，但三个模块的 `steps` 和 `conclusion` 的具体内容不同：

**Plausible Reasoning 超边：**
```json
{
    "premises": ["三角形ABC", "M是AB中点", "N是AC中点"],
    "steps": [
        "M和N分别是两边中点，连接MN形成截线",
        "由相似三角形的性质，三角形AMN与三角形ABC相似",
        "相似比为1:2，因此MN应该平行于BC且长度为BC的一半"
    ],
    "conclusion": "MN平行于BC且MN=BC/2",
    "module": "plausible",
    "confidence": 0.6
}
```

**Experiment 超边：**
```json
{
    "premises": ["三角形ABC", "M是AB中点", "N是AC中点"],
    "steps": [
        "# 代码或代码patch",
        "A=(0,4), B=(-3,0), C=(5,0) → MN=4.0, BC=8.0, ratio=0.5",
        "A=(1,7), B=(0,0), C=(6,0) → MN=3.0, BC=6.0, ratio=0.5",
        "A=(2,3), B=(-1,1), C=(5,-2) → ratio=0.5",
        "100次随机测试: ratio始终为0.500000 (误差<1e-12)",
        "斜率检测: MN与BC斜率在所有测试中完全一致"
    ],
    "conclusion": "数值证据强烈支持: MN∥BC 且 MN=BC/2",
    "module": "experiment",
    "confidence": 0.92
}
```

**Lean 超边：**
```json
{
    "premises": ["三角形ABC", "M是AB中点", "N是AC中点"],
    "steps": [
        "theorem midline_parallel (A B C : Point) :",
        "  let M := midpoint A B",
        "  let N := midpoint A C",
        "  parallel (line_through M N) (line_through B C) := by",
        "    <tactic proof>..."
    ],
    "conclusion": "midline_parallel: ∀ A B C, ... → parallel(MN, BC)",
    "module": "lean",
    "confidence": 0.99
}
```

### 3.3 命题节点

超图中的节点是**命题 (proposition)**。每个命题节点包含：

```
命题节点 (Node) = {
    id:           唯一标识符,
    statement:    命题内容（自然语言 + 可选的形式化表述）,
    belief:       float ∈ [0, 1],    -- 当前置信度（由贝叶斯推断计算）
    provenance:   [hyperedge_ids],   -- 所有指向此节点的超边
    discovered_at: timestamp,
    domain:       "geometry" | "number_theory" | ...
}
```

### 3.4 超图的增长

系统运行过程中，超图持续增长：
- 每次 plausible reasoning 产出新猜想 → 新节点 + 新超边
- 每次实验验证一个猜想 → 新超边指向已有节点（增强置信度）
- 每次 Lean 证明成功 → 新超边指向已有节点（置信度坍缩至 ~1.0）
- 新的定理可以作为后续推理的前提 → 超图形成多层结构

## 4. 贝叶斯推断：从局部可靠性到全局置信度

### 4.1 条件概率与 LLM-as-Judge

每条超边都附带一个**条件概率** `confidence`：

> P(conclusion 正确 | 所有 premises 都正确)

这个概率由 **LLM-as-Judge** 模块估计。Judge 在每条超边生成后，独立评估其推理步骤的可靠性并打分。

### 4.2 置信度传播

给定超图结构和每条超边的条件概率，可以计算每个命题节点的**全局置信度**。

**基本规则：**

- 公理节点：belief = 1.0（无条件为真）
- 对于有多条超边指向的节点，每条路径提供独立证据
- 置信度通过贝叶斯规则在超图上传播

**多路径增强：**

同一个命题如果被多条独立路径支持，置信度叠加：

| 路径 | 来源 | confidence |
|---|---|---|
| 路径1 | plausible reasoning | 0.5 |
| 路径2 | 数值实验 100 例 | 0.9 |
| 路径3 | 类比另一个已知定理 | 0.4 |
| **组合后** | **多路径贝叶斯融合** | **显著高于任何单条路径** |

**Lean 证明的坍缩效应：**

一旦某个节点获得 Lean 证明（confidence ~1.0），其 belief 直接坍缩为 ~1.0，不再依赖其他路径。这使得 Lean 证明成为"终极确认"。

### 4.3 不确定性的反向传播

如果一个高置信度节点的某个前提被发现有问题（例如找到反例），不确定性会沿超图**反向传播**，自动降低所有依赖该前提的下游节点的置信度。

## 5. 探索策略：Active Inference

### 5.1 核心决策问题

在每一步，系统需要决定：
1. **探索什么领域？** （几何、数论、代数……）
2. **构造什么对象？** （新的点、新的数列、新的群……）
3. **使用哪个模块？** （plausible reasoning、实验、Lean）
4. **针对哪个节点？** （哪个猜想最值得投入资源）

### 5.2 价值-不确定性权衡

系统通过以下启发式选择下一步行动：

```
priority(节点) = value(节点) × (1 - belief(节点))
```

- **value 高 × belief 低** → 最值得投入（高价值但不确定的猜想）
- **value 高 × belief 高** → 已经基本确认，不需要更多资源
- **value 低 × belief 低** → 不值得投入
- **value 低 × belief 高** → 已确认的平凡事实

其中 `value` 可以综合考虑：
- 证明长度的倒数（简洁的定理更有价值）
- 结构新颖性（连接不同概念的定理更有价值）
- 复用潜力（被其他证明引用的潜力）
- 所属领域的探索深度（鼓励在新领域做探索）

### 5.3 模块选择策略

针对一个选定的目标节点，选择使用哪个模块：

- 当前 belief 很低、还没有任何证据 → **plausible reasoning**（先形成初步判断）
- 有了初步猜想但不确定 → **experiment**（用数值实验快速验证）
- 实验证据已经很强 → **Lean proof**（投入资源做形式化证明）
- Lean 证明卡住了 → 退回 **plausible reasoning**（换个证明策略）

这不是固定流水线——LLM 根据当前超图状态自主决定，就像数学家灵活选择工作模式一样。

## 6. 系统架构

### 6.1 三层架构

**探索策略层** — "下一步该做什么？"

- 超图状态分析
- 贝叶斯置信度传播
- 价值-不确定性计算
- 行动选择 (探索方向 + 目标节点 + 模块选择)

**Skill 层** — "怎么做这一步？"

| Skill | 功能 |
|---|---|
| Plausible Reasoning | LLM 非形式推理 --> 超边(premises, 推理链, conclusion) |
| Experiment | LLM 写代码 + 执行 --> 超边(premises, 代码+结果, conclusion) |
| Lean Proof | LLM 写 Lean tactic + 验证 --> 超边(premises, proof, conclusion) |
| Judge | LLM 评估超边可靠性 --> confidence 打分 |

**Agent 框架层** — "执行具体操作"

基于 Claude Code / OpenCode 等工业级 Agent 框架:
- Shell 交互 (调用 Lean, 执行计算脚本)
- 文件管理 (读写 Lean 文件, 实验代码)
- Patch 管理 (代码修改, 回滚)
- 错误恢复 (命令失败处理, 重试逻辑)

### 6.2 四个 Skill 的详细设计

#### Skill 1: Plausible Reasoning

**触发条件：** LLM 决定对某个方向进行非形式推理。

**输入：** 当前超图上下文 + 目标方向描述。

**过程：**
1. LLM 选择一组前提（从超图中的已有节点或新构造）
2. 进行自然语言推理（类比、归纳、模式识别、联想）
3. 得出结论（一个新的猜想或对已有猜想的新理解）

**输出：** 一条超边，`module = "plausible"`。

**特点：** 这是唯一不依赖外部工具的 skill——本质就是 LLM 自身的推理能力。

#### Skill 2: Experiment

**触发条件：** LLM 决定用数值实验验证某个猜想，或在某个领域做探索性实验。

**输入：** 待验证的猜想 + 相关数学对象的定义。

**过程：**
1. LLM 编写实验代码（Python / SageMath / 其他）
2. 通过 Agent 框架执行代码
3. LLM 解读计算结果
4. 判断实验是否支持猜想

**输出：** 一条超边，`module = "experiment"`，steps 包含代码和计算结果。

**特点：** 实验可以是验证性的（检验已有猜想）或探索性的（在新领域寻找规律）。

#### Skill 3: Lean Proof

**触发条件：** LLM 决定对某个猜想进行形式化证明。

**输入：** 待证命题 + 可用的 Mathlib 定理 + 已有定理库。

**过程：**
1. LLM 编写 Lean 4 证明代码
2. 通过 Agent 框架调用 Lean 验证
3. 如果失败，分析错误信息并迭代修改
4. 证明通过后，提取形式化的定理声明

**输出：** 一条超边，`module = "lean"`，confidence ~1.0。

**特点：** 这是最昂贵但最可靠的模块。成功的 Lean 证明是定理的终极确认。

#### Skill 4: Judge

**触发条件：** 每条超边生成后自动触发。

**输入：** 一条完整的超边（premises + steps + conclusion）。

**过程：**
1. LLM 独立审视推理步骤的每一步
2. 评估逻辑跳跃是否合理
3. 考虑可能的反例或漏洞
4. 给出条件概率估计

**输出：** 更新超边的 `confidence` 字段。

**特点：** Judge 应使用不同的 prompt/视角，避免与生成超边的推理过程产生确认偏差。

### 6.3 数据流

以"三角形三条中线交于一点"为例，展示完整的探索过程：

**Step 1 -- Plausible Reasoning**

- 探索策略: "探索三角形中线性质, 使用 plausible reasoning"
- 输入: [三角形ABC, 三条中线]
- 输出: 超边 --> "三条中线可能交于一点" (猜想)
- Judge 打分: confidence = 0.55
- 超图更新: 新节点 "三条中线交于一点" (belief = 0.55)

**Step 2 -- Experiment**

- 探索策略: "这个猜想 value 高、belief 中等 --> 值得做实验"
- 输入: "验证三角形三条中线是否交于一点"
- 输出: 超边 --> "1000次随机测试全部通过"
- Judge 打分: confidence = 0.93
- 超图更新: belief("三条中线交于一点") 显著提升

**Step 3 -- Lean Proof**

- 探索策略: "belief 已经很高 --> 值得投入 Lean 证明"
- 输入: "证明: forall 三角形, 三条中线交于一点"
- 输出: 超边 --> Lean proof term
- 超图更新: belief --> ~1.0, 定理入库

## 7. 技术栈

### 7.1 核心组件

| 组件 | 技术选型 | 说明 |
|---|---|---|
| Agent 框架 | Claude Code / OpenCode | 提供 Shell、文件管理、工具交互基础设施 |
| LLM | Claude / GPT / 可配置 | 核心推理引擎，可根据需要切换模型 |
| 形式化验证 | Lean 4 + Mathlib | 严格的数学证明验证 |
| 数值计算 | Python + NumPy/SageMath | 实验验证和探索性计算 |
| 超图存储 | 自定义数据结构 + 持久化 (SQLite/JSON) | 管理命题节点和推理超边 |
| 贝叶斯推断 | 自定义模块 | 在超图上传播置信度 |

### 7.2 Lean 交互方式

系统通过 Agent 框架的 Shell 能力与 Lean 交互：
- 将待证命题写入 `.lean` 文件
- 调用 `lake build` 或 `lean` 命令进行验证
- 解析输出（成功 / 错误信息）
- 也可集成 LeanDojo 或 lean4-repl 以获得更细粒度的交互

### 7.3 Skill 实现

每个 Skill 作为 Agent 框架的标准 skill 实现：
- 定义输入输出格式（统一的超边格式）
- 包含引导 LLM 的 prompt 模板
- 包含工具调用的工作流定义
- LLM 按需调用，不是固定流水线

## 8. 探索循环的完整流程

系统运行时不存在固定的"一局游戏"。它是一个**持续运行**的探索过程：

**初始化:** 超图 = {公理节点集合} (来自 Lean/Mathlib 的基础公理)

**持续循环:**

1. **策略分析** -- 探索策略层分析超图当前状态
   - 计算所有节点的 value x (1 - belief)
   - 识别高优先级目标
   - 决定探索方向和使用的 skill

2. **执行推理** -- 调用选定的 skill 执行一步推理
   - Skill 产出一条超边
   - Judge 对超边打分

3. **超图更新** -- 整合新知识
   - 添加新节点和超边
   - 贝叶斯推断重新计算受影响节点的 belief
   - 已证明定理加入定理库 (可在后续推理中作为前提使用)

4. **定理提取** -- 每一步推理都可能产出新定理
   - 每条推理动作自动提取为自包含命题
   - 查重: 是否已在库中? (模变量重命名等价)
   - 如果是新的, 加入库 + 奖励信号
   - 奖励大小 = f(证明长度倒数, 结构新颖性, 复用潜力)

5. 回到步骤 1, 继续探索

### 8.1 新定理的定义

一条命题成为**新定理**需要满足：
1. 被系统成功推导出（通过 plausible + experiment + lean 中的任意组合）
2. 是普遍性命题（对所有满足条件的数学对象成立）
3. 不在已有定理库中（模 alpha 等价和变量重命名）
4. 不是任何构造步骤的直接定义性质

定理的**价值**不需要人为划定"平凡 vs 非平凡"的边界，而是由奖励函数连续评估。

### 8.2 定理的自动提取

当推理步骤产出新事实时，系统自动进行定理提取：
1. 收集结论所依赖的所有前提和构造步骤
2. 将构造步骤内联，打包为自包含的陈述
3. 对自由变量做全称量化
4. 与定理库中已有条目做等价性检查
5. 新定理入库，同时作为后续推理可用的前提

## 9. 长期愿景

### 9.1 分阶段推进

**Phase 1: 基础设施搭建**
- 实现超图数据结构和基本的贝叶斯推断
- 实现四个 skill 的基本版本
- 在几何领域做端到端验证（利用数值检测的优势）

**Phase 2: 能力验证**
- 验证系统能否重新发现已知的经典几何定理
- 评估系统发现新关系的能力
- 优化探索策略

**Phase 3: 领域扩展**
- 扩展到数论、组合等其他数学领域
- 针对不同领域调优实验模块

**Phase 4: 体系优化**
- 在积累足够多定理后，分析定理之间的依赖关系
- 寻找更高效的公理化路径（更少的引理覆盖更多的定理）
- 这是最初的"比人类体系更高效"愿景的实现

### 9.2 成功标准

- 能够在选定领域内自主重新发现经典定理
- 能够发现至少一个人类定理体系中未被显式陈述的新关系
- 证明体系的"压缩率"优于传统教科书的组织方式
- 系统能够跨领域迁移探索能力（在新领域无需重新设计即可开始发现）

## 附录 A: 关键术语

| 术语 | 定义 |
|---|---|
| 超图 (Hypergraph) | 允许一条边连接多个节点的图。这里用于表示"多个前提推出一个结论" |
| 超边 (Hyperedge) | 超图中的一条边，表示一步推理：{premises} → conclusion |
| 置信度 (Belief) | 一个命题节点为真的概率估计，由贝叶斯推断计算 |
| 条件概率 (Confidence) | 一条超边的可靠性：P(conclusion正确 \| premises都正确) |
| Plausible Reasoning | Polya 意义上的合情推理：类比、归纳、直觉等非严格但有启发性的推理方式 |
| Active Inference | 根据当前信念状态（belief state）选择最有信息价值的下一步行动 |
| 定理提取 | 从推理过程中自动抽取自包含的、可复用的定理声明 |
